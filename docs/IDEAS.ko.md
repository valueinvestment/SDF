# 아이디어 및 향후 방향
# SDF 디지털 트윈 멀티 에이전트 시뮬레이터

아이디어는 카테고리별로 분류되며 예상 노력과 임팩트로 태그됩니다.

**노력:** S (1일 미만) · M (1~3일) · L (1~2주) · XL (2주 이상)  
**임팩트:** 낮음 · 중간 · 높음 · 쇼케이스 (포트폴리오 핵심)

---

## 시각화

### V-01: 기계 유형별 3D 모델
**노력:** M · **임팩트:** 높음  
균일한 박스 지오메트리를 기계 유형별 구별된 형태로 교체: 납작한 프레스(넓고 평평한 박스), CNC 선반(하우징이 있는 실린더), 컨베이어(벨트 텍스처의 긴 얇은 박스). 복잡성 추가 없이 시각적 사실성을 크게 향상.

### V-02: 로봇 이동 경로 시각화
**노력:** S · **임팩트:** 중간  
`THREE.Line`과 시간에 따라 불투명도가 감소하는 커스텀 셰이더를 사용하여 파견된 로봇 뒤에 페이딩 라인 트레일 렌더링. 파견 경로를 시각적으로 명확하게 표현.

### V-03: 고장 발생 시 파티클 이펙트
**노력:** M · **임팩트:** 높음  
기계가 고장 상태에 진입할 때, `THREE.Points`와 커스텀 파티클 머티리얼을 사용하여 파티클 시스템(불꽃/연기) 방출. 문제가 발생했다는 즉각적인 시각적 신호 생성. 고장 해제 시 파티클 dispose.

### V-04: 아이소메트릭 카메라 전환
**노력:** S · **임팩트:** 중간  
원근 카메라(3D 몰입형)와 직교 카메라(아이소메트릭, 탑다운) 전환 버튼 추가. 직교 뷰는 비기술 직군 리크루터가 읽기 더 쉬움.

### V-05: WebGL 인스턴스 렌더링
**노력:** L · **임팩트:** 중간 (V2 스케일에 필요)  
기계와 로봇에 대해 개별 `THREE.Mesh` 객체를 `THREE.InstancedMesh`로 교체. 드로우 콜을 유형당 O(n)에서 O(1)로 감소. FPS 저하 없이 50대 이상 기계로 확장하는 데 필수.

### V-06: 바닥 텍스처
**노력:** S · **임팩트:** 중간  
`GridHelper`를 콘크리트 바닥 텍스처를 사용한 텍스처 평면으로 교체. 미묘한 범프 매핑 추가. 30분 미만의 작업으로 시각적 깊이 추가.

### V-07: 실시간 히트맵 오버레이
**노력:** M · **임팩트:** 쇼케이스  
고장 기계 근접도에 따른 "위험 구역"을 보여주는 WebGL 히트맵 텍스처를 공장 바닥에 렌더링. 센서 틱마다 `DataTexture`를 통해 텍스처 업데이트. 시각적으로 인상적이고 기술적으로 흥미로움.

## AI 및 에이전트 지능

### A-01: 컨텍스트 윈도우를 통한 에이전트 메모리
**노력:** M · **임팩트:** 높음  
최근 3개의 사건 보고서를 각각의 새 에이전트 A 호출에 컨텍스트로 전달. 진단 에이전트가 동일 기계의 반복 고장을 인식하고 "만성 저하" vs. 일회성 고장으로 분류 가능. 더 나은 AI 출력, 최소한의 엔지니어링.

### A-02: 스트리밍 에이전트 응답
**노력:** M · **임팩트:** 쇼케이스  
`anthropic.messages.stream()`을 사용하고 에이전트가 "생각하는" 동안 토큰 청크를 WebSocket으로 전달. Agent Panel에서 에이전트 출력을 스트리밍 텍스트로 렌더링 — "AI가 실시간으로 타이핑하는" 효과. 포트폴리오의 시각적 하이라이트 순간.

### A-03: 에이전트 D — 예측 정비 스케줄러
**노력:** L · **임팩트:** 쇼케이스  
5분 타이머로 실행되는 네 번째 에이전트 추가(고장으로 트리거되지 않음). 진동 트렌드 기울기를 기반으로 모든 기계의 센서 트렌드 이력을 검토하고, 다음에 고장날 기계와 시기를 예측. 출력: 긴급도순으로 정렬된 정비 일정.

### A-04: 피드백 루프 — 에이전트 결과 추적
**노력:** M · **임팩트:** 높음  
에이전트 B가 로봇을 파견하고 "도착"한 후, `repair_complete` 이벤트 발행. 동일 기계에 대한 에이전트 C의 다음 실행 시 수리 결과를 컨텍스트로 받아, 시간에 따른 RICE 점수 정제 가능(시뮬레이션 학습 루프).

### A-05: 멀티 모델 비교 패널
**노력:** L · **임팩트:** 높음  
`claude-haiku-4-5`와 `claude-sonnet-4-6` 모두에서 에이전트 A를 동시 실행하고 UI에 결과를 나란히 표시. 라이브 데모 기능으로 모델 성능 비교 제공 — AI 엔지니어링 직군에 직접적으로 관련.

### A-06: 에이전트 B에 도구 사용(Tool Use) 적용
**노력:** L · **임팩트:** 높음  
에이전트 B를 프롬프트 기반 JSON 출력 대신 Claude의 도구 사용(함수 호출)으로 리팩토링. `select_robot(robot_id)`, `plan_path(start, end)` 같은 도구 정의. 더 견고한 파싱, 더 진정성 있는 "에이전틱" 동작, 도구 사용 숙련도 증명.

### A-07: 에이전트 신뢰도 임계값 조정
**노력:** S · **임팩트:** 중간  
에이전트 A가 체인을 트리거하는 최소 신뢰도 임계값을 제어하는 UI 슬라이더 추가. 에이전트 A가 임계값 미만의 신뢰도를 반환하면 B와 C를 실행하는 대신 "모니터링" 상태를 발행. 설정 가능한 AI 민감도 증명.

## 데이터 파이프라인

### D-01: 백프레셔 메트릭 대시보드
**노력:** M · **임팩트:** 중간  
프론트엔드에서 WebSocket 메시지 큐 깊이를 추적하고 작은 인디케이터로 표시. 큐 깊이가 50을 초과하면 "따라잡는 중" 배지 표시. 실시간 시스템 상태 인식 증명 — 좋은 기술 토론 포인트.

### D-02: 시뮬레이션된 OPC-UA 네임스페이스
**노력:** L · **임팩트:** 높음  
커스텀 센서 형식을 OPC-UA 노드 ID(`ns=2;s=Machine1.Vibration`)로 교체하고 센서 데이터를 OPC-UA 주소 공간으로 구조화. 순수 시뮬레이션 — 실제 OPC-UA 서버 없음 — 이지만 산업 프로토콜에 대한 도메인 지식 증명.

### D-03: 상태 동기화를 포함한 WebSocket 재연결
**노력:** M · **임팩트:** 중간  
재연결 시, 프론트엔드가 `GET /state` REST 엔드포인트를 통해 "상태 스냅샷"을 요청하여 Zustand 스토어를 즉시 재수화. 다음 센서 틱을 기다리는 대신 재연결 후 빈 차트를 방지.

### D-04: 멀티 인스턴스 백엔드를 위한 Redis Pub/Sub
**노력:** L · **임팩트:** 중간  
인프로세스 `EventBus`를 Redis pub/sub으로 교체. 백엔드의 수평 확장 가능(로드 밸런서 뒤의 여러 Railway 인스턴스). 프로덕션 스케일에 필요, 좋은 아키텍처 토론 포인트.

### D-05: 서버 사이드 다운샘플링
**노력:** M · **임팩트:** 중간  
ECharts 이력 뷰에서 최근 30초 데이터만 전체 해상도로 전송. 30초 이상 된 데이터(이력 엔드포인트가 추가된 경우)는 LTTB(Largest Triangle Three Buckets) 알고리즘을 사용하여 1Hz로 다운샘플링 후 전송. 표준 시계열 최적화 기법.

## 사용자 경험

### U-01: 기계 상세 패널 (클릭하여 검사)
**노력:** M · **임팩트:** 높음  
Three.js에서 기계 메시를 클릭하면 세 가지 센서 값(진동, 온도, 전류)을 각각 ECharts 게이지로 보여주는 상세 패널 열기. Three.js에서의 레이캐스팅과 더 풍부한 센서 데이터 시각화 증명.

### U-02: 속도 제어 (시뮬레이션 시간 배율)
**노력:** S · **임팩트:** 중간  
시뮬레이터 틱 속도를 배수하고 고장 주입 간격을 비례적으로 압축하는 UI 컨트롤(1x / 5x / 10x) 추가. 방문자가 90초 대신 10초 내에 에이전트 체인을 트리거할 수 있게 함. 라이브 데모에 필수적.

### U-03: 사건 이력 로그
**노력:** M · **임팩트:** 중간  
타임스탬프, 기계, 고장 분류, RICE 권고사항이 있는 모든 과거 사건의 스크롤 가능한 사이드바 로그. 데모에 깊이 추가 — 이것이 단순한 시각화 이상임을 보여줌.

### U-04: 다크 / 산업 테마 전환
**노력:** S · **임팩트:** 낮음  
다크 `gray-950` 배경과 흰색 배경 + 파란색 악센트의 "산업" 테마 사이 전환 토글 추가. 최소한의 작업, UI 폴리시 인식 증명.

### U-05: 에이전트 보고서 PDF 내보내기
**노력:** M · **임팩트:** 중간  
`@react-pdf/renderer`를 사용하여 RICE 보고서 PDF를 생성하는 Agent Panel의 "보고서 다운로드" 버튼. AI 출력이 단순한 UI 텍스트가 아닌 실행 가능한 문서임을 증명.

## 로보틱스 및 경로 탐색

### R-01: A* 경로 탐색
**노력:** L · **임팩트:** 높음  
에이전트 B의 직선 경유지를 공장 격자에서의 실제 A* 구현으로 교체. 로봇이 기계 장애물을 통과하는 대신 피해서 이동. 프론트엔드(파견 시 경로 탐색)의 TypeScript 또는 백엔드의 Python으로 구현.

### R-02: 로봇 충돌 회피
**노력:** XL · **임팩트:** 쇼케이스  
두 로봇이 동시에 파견될 때, 속도 장애물(VO) 또는 상호 속도 장애물(RVO) 알고리즘을 사용한 충돌 회피 구현. 로봇이 서로를 동적으로 우회. 기술적으로 까다롭지만 시각적으로 인상적.

### R-03: 로봇 유휴 애니메이션
**노력:** S · **임팩트:** 중간  
로봇이 유휴 상태일 때 느린 회전 애니메이션(제자리 회전). 파견되면 회전이 멈추고 이동 시작. 30분 미만의 작업으로 씬에 생동감 추가.

### R-04: 멀티 로봇 파견
**노력:** M · **임팩트:** 높음  
에이전트 B는 현재 로봇 1대를 파견. 높은 심각도의 고장에 대해 로봇 2대를 파견하도록 확장(1대는 수리, 1대는 부품 배달). `DispatchCommand` 스키마를 배열 지원으로 업데이트하고 두 개의 애니메이션 경로 조율 필요.

## 인프라 및 개발자 경험

**노력:** S · **임팩트:** 중간  
단일 `docker compose up`으로 프론트엔드와 백엔드를 모두 시작하는 `docker-compose.yml` 추가. 레포를 클론하는 누구에게나 온보딩 간소화.

### I-02: GitHub Actions CI
**노력:** S · **임팩트:** 중간  
`main`에 푸시할 때마다 `pytest`(백엔드)와 `vitest`(프론트엔드)를 실행하는 워크플로우. 레포에서 전문적인 엔지니어링 습관 증명.

### I-03: Playwright E2E 테스트
**노력:** M · **임팩트:** 높음  
페이지 로드 → 3D 캔버스 렌더링 대기 → 테스트 API 엔드포인트를 통한 고장 주입 → AlertBanner 표시 대기 → Agent Panel이 세 개의 완료 이벤트를 보여주는 것을 어서션하는 Playwright 테스트. E2E 테스팅 지식 증명.

### I-04: UI 컴포넌트 Storybook
**노력:** M · **임팩트:** 낮음  
모킹된 데이터로 `AgentPanel`, `AlertBanner`, `SensorChart`에 대한 Storybook 스토리 추가. 컴포넌트 API 문서화 및 컴포넌트 주도 개발 실천 증명.

### I-05: WebSocket 부하 테스트
**노력:** S · **임팩트:** 중간  
`asyncio`를 사용하여 백엔드에 50개의 동시 WebSocket 연결을 열고 클라이언트별 메시지 전달 레이턴시를 측정하는 Python 스크립트. 시스템 용량 한계 문서화.
